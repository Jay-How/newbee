彼岸史诗：战团 战斗规则说明书
一、游戏基本信息
1.	游戏名称：彼岸史诗：战团
2.	游戏人数：单人体验版（Demo），多人模式待后续开发
3.	游戏胜利目标：玩家作为远征队指挥官，构筑卡组、指挥战团护送圣物，借助圣物能量击败所有敌人与最终BOSS。
4.	游戏配件：随机生成式肉鸽地图系统、玩家卡牌（单位卡、策略卡）、敌方卡牌、圣物
二、游戏准备
1. 卡组构筑
•	构筑规则：无任何限制条件，卡牌数量、卡牌类型组合均不设约束，支持玩家灵活构筑个性化卡组
•	初始手牌：初始手牌配置将后续补充示例方案，供玩家选择。
2. 场地布置
•	玩家区域：分为前排区域与后排区域，默认每区域最大部署数量为4张单位卡；可通过遗物、装备等增益效果提升单区域部署上限
•	核心布置：玩家后排区域需放置圣物，圣物为游戏失败判定的核心目标
•	敌方区域：单排区域，无前后排区分
3. 起始状态设定
•	圣物初始能量值：默认10点
•	行动点初始上限：默认4点
三、回合流程
1. 回合开始阶段
•	行动点重置：玩家行动点补充至当前上限（初始上限为4点）
•	法强值结算：统计当前法强值，计算公式为「法强值 = 圣物剩余能量值 + 已激活单位法力值总和」
2. 主要阶段
可执行以下核心操作，操作均无次数限制（特殊规则除外）：
•	单位位移：含移动、前后排换位，默认不消耗行动点；位移后单位本回合进入冷却状态，无法使用技能（激活单位不触发冷却）
•	付费抽牌：消耗行动点抽牌，抽中卡牌自动置入己方后排
•	单位激活：消耗行动点（消耗量=单位法力数值）执行激活；激活后不触发冷却，状态持续保持（特殊情况除外）
•	技能发动：消耗行动点（消耗量以单位卡技能费用为准）发动已激活单位的技能；需满足区域使用限制（如单位技能默认前排使用）
•	策略卡发动：按卡牌描述消耗对应资源完成发动（主动发动类仅可在此阶段发动），发动后效果可在任意回合阶段执行
3. 回合结束阶段
执行回合收尾判定与机制触发，包括但不限于：
•	持续效果结算：增益/减益效果存续判定、回合性效果触发
•	特殊状态校验：单位受伤状态后续判定等
•	被动策略卡触发：满足费用预留条件的被动策略卡在此阶段发动
•	敌方回合前置准备：触发敌方回合前置准备机制（具体以敌方卡牌规则为准）
四、卡牌类型与规则
1. 单位卡
卡牌构成：单位名称、血量、法力、激活费用、单位技能、技能描述、技能费用
•	费用规则：所有费用（激活费用、技能费用）均消耗行动点
•	激活费用 = 单位法力数值
核心使用规则：
•	基础前提：单位必须激活后才能使用技能
•	激活状态：激活后持续保持，除非被怪物打断等特殊情况触发，否则不解除
•	操作时机：激活与技能使用均仅可在主要阶段执行（特殊规则除外）
•	费用限制：行动点不足时，无法激活或使用技能（特殊说明、专属能力或道具加持除外）
•	次数限制：回合内激活次数、技能使用次数均无限制（可根据后续需求补充限制）
部署与存活规则：
•	技能使用区域：除非有特殊说明，单位技能仅可在前排使用
•	移动冷却：单位从后排移至前排，或与其他单位执行前后排位置互换后，本回合进入冷却状态，无法使用技能；激活单位后不会触发冷却状态
•	受伤与退场：单位血量降至1点时，触发受伤状态并强制退场，进入弃牌堆
•	死亡判定：血量归零的单位，判定为死亡，直接进入墓地
•	牌堆循环：抽牌堆耗尽时，弃牌堆将被洗匀成为新的抽牌堆；弃牌堆中的单位卡牌可重新部署，但部署后血量不自动恢复（特殊情况说明除外）
费用规则：
•	激活费用 = 消耗单位法力值相等的行动点，
•	技能费用 = 消耗一定行动点和法强
2. 策略卡
卡牌构成：策略名称、策略类型（战术卡/法术卡）、发动类型（主动发动/被动发动）、使用费用、使用效果
核心使用规则：
•	发动类型区分：策略卡分为主动发动与被动发动两类，两类卡牌的使用时机、费用要求存在差异
•	主动发动规则：
① 使用时机：仅限主要阶段发动（特殊规则除外）；
② 费用消耗：战术卡仅消耗对应行动点（具体以卡牌描述为准），不消耗法强；法术卡需同时消耗对应行动点与指定法强值（具体消耗量以卡牌描述为准）
•	被动发动规则：
① 使用时机：在卡牌描述的阶段发动（一般是敌方回合）；
② 费用约束：需在回合结束阶段预留对应使用费用（战术卡预留行动点、法术卡预留行动点+法强），未预留则无法触发；
③ 效果执行：发动后效果可按卡牌描述在指定回合阶段执行，执行顺序默认按时间顺序。
3. 敌方卡牌
卡牌核心构成
•	基础属性：敌方名称、等级、血量、法强，仇恨对象
•	技能相关：专属技能、技能触发条件
•	行动相关：行动优先级、专属意图池（含各意图概率权重）
核心行动规则
•	行动时机：紧随玩家回合结束阶段，完成敌方前置准备后启动
•	行动顺序：按部署编号从小到大执行；标注“优先行动”词条的单位无视编号优先行动
•	意图-概率判定机制：
① 意图判定：行动前通过骰子（桌游）/随机生成（电子游戏）确定本次意图；
② 执行规则：校验意图触发条件，达成则执行，未达成再次随机；
③ 冷却机制：技能类意图触发后进入冷却
•	目标锁定：按仇恨优先级锁定目标（普通敌方单目标，BOSS可多目标），无仇恨目标时默认攻击圣物
部署规则
•	部署区域：专属单排作战区域，与玩家前排呈对战态势
•	部署时机与数量：基础/精英阶段战斗开始时一次性部署（上限3/5张）；BOSS阶段先部署2张辅助卡，辅助卡存活时再部署1张BOSS卡
•	部署后状态：立即解锁意图池，无需激活即可参与下一轮行动判定
•	补充部署：战斗中可通过召唤类意图生成新单位（置入区域末端，编号顺延，生成回合即可行动）
存活与退场规则
•	死亡判定：血量降至0判定死亡，进入敌方墓地不可复用
•	持续状态：无“受伤状态”，血量未归零前持续参与行动
•	战斗结束：所有敌方单位死亡时，当前战斗阶段结束，进入战利品结算
4. 通用卡牌规则：
•	使用时机约束：卡牌仅可在对应合法时机发动（单位卡、策略卡具体时机以各自规则为准），违规发动视为无效，不消耗任何资源
•	区域置入规则：卡牌发动后需按规则置入对应区域（单位卡置入作战区域、策略卡发动后置入弃牌堆），不可重复使用
•	使用次数限制：单回合内卡牌使用次数无默认限制，仅受特殊卡牌描述、资源上限及阶段规则约束
•	效果冲突处理：卡牌效果冲突时，优先执行“特殊说明”条款；无特殊说明则按“后发动效果覆盖先发动效果”原则处理
•	未激活单位限制：未激活的单位卡仅可执行位移操作，不可发动技能及其他主动效果
•	魔法技能使用约束：任何一方总法强≤对方总法强时，该方不可对对方使用带有“魔法”词条的技能
五、战斗核心规则
1. 基础部署与移动机制
•	部署区域：玩家单位需部署于前排/后排指定作战区域，敌方单位部署于单排作战区域
•	位移操作：含前后排换位、同排位置调整，默认不消耗行动点（特殊卡牌效果除外）；位移后单位本回合进入冷却状态，无法使用技能；激活单位不会触发冷却
•	部署上限：玩家每区域默认最大部署4张单位卡，可通过增益效果提升上限
2. 技能使用核心规则
•	首回合特殊规则：需消耗1点行动点解锁技能使用权限，未解锁则本回合无法发动任何卡牌技能；可正常激活卡牌，激活后不触发冷却
•	通常回合规则：仅已激活单位可发动技能；技能发动需消耗对应行动点（法术类技能额外消耗法强）；需满足区域使用限制（如单位技能默认前排使用）
3. 攻击与防御交互
攻击规则：
•	攻击类型划分：攻击分为物理攻击、魔法攻击两大类，每类均包含普通攻击、特殊攻击两个子类型，攻击类型与伤害类型严格匹配（物理攻击对应物理伤害，魔法攻击对应魔法伤害）
各攻击类型核心规则：
•	普通攻击：成功命中后造成对应类型普通伤害，伤害数值以攻击方基础属性标注为准
•	特殊攻击：成功命中后造成对应类型特殊伤害，伤害效果按目标血量差异化判定
伤害效果结算：
•	普通伤害：直接减少目标血量值
•	特殊伤害：目标血量＞1时，血量直接降至1并强制退场进入弃牌堆；目标血量＝1时，直接判定死亡
•	通用结算：普通攻击、特殊攻击命中血量＝1的目标时，均直接判定目标死亡并进入墓地

# 彼岸史诗：战团 简单Demo架构设计

## 一、架构设计原则

1. **简化优先**：采用最小化设计，仅实现核心游戏机制，确保快速原型验证。
2. **模块化**：将游戏逻辑拆分为独立模块，便于后续扩展和修改。
3. **数据驱动**：使用配置文件定义卡牌、敌人等数据，便于快速调整游戏平衡。
4. **可扩展性**：预留接口，为后续功能迭代做准备。

## 二、核心模块设计

### 2.1 游戏核心管理器（GameManager）

**功能**：作为游戏的核心枢纽，管理游戏状态和模块间通信。

**实现**：
- 采用简化的状态机，仅包含必要状态：MenuState（菜单）、BattleState（战斗）、EndState（结束）
- 负责场景切换和模块初始化
- 处理全局事件和异常

### 2.2 卡牌管理器（CardManager）

**功能**：管理卡牌的加载、使用和效果结算。

**实现**：
- 从配置文件加载卡牌数据
- 管理卡组、手牌、抽牌堆、弃牌堆
- 处理卡牌使用逻辑和效果触发

### 2.3 战斗管理器（BattleManager）

**功能**：管理战斗逻辑和回合流程。

**实现**：
- 实现回合流程：回合开始→主要阶段→回合结束
- 管理行动点和法强值
- 处理单位位移、激活、技能发动等操作
- 管理敌方AI行为

### 2.4 敌人管理器（EnemyManager）

**功能**：管理敌人的部署和行为。

**实现**：
- 从配置文件加载敌人数据
- 实现敌人的意图-概率判定机制
- 处理敌人的行动和技能

### 2.5 UI管理器（UIManager）

**功能**：管理游戏界面的显示和交互。

**实现**：
- 实现主菜单、战斗界面、结束界面
- 处理卡牌显示、行动点显示、圣物能量显示
- 处理用户输入和交互

### 2.6 数据管理器（DataManager）

**功能**：管理游戏数据的存储和读取。

**实现**：
- 使用Godot的Resource类实现数据资源的存储、取用和序列化
- 管理游戏状态的保存和加载
- 处理数据验证和错误处理

## 三、数据结构设计

### 3.1 卡牌数据结构（使用Resource类）

```gdscript
# CardResource.gd
class_name CardResource
extends Resource

@export var card_id: String
@export var card_name: String
@export var card_type: String
@export var cost: int
@export var attack: int
@export var health: int
@export var mana: int
@export var skill: Dictionary
```

**资源文件示例**：`res://resources/cards/card_001.tres`

### 3.2 敌人数据结构（使用Resource类）

```gdscript
# EnemyResource.gd
class_name EnemyResource
extends Resource

@export var enemy_id: String
@export var name: String
@export var level: int
@export var health: int
@export var mana: int
@export var intent_pool: Array
```

**资源文件示例**：`res://resources/enemies/enemy_001.tres`

### 3.3 游戏状态数据结构

```gdscript
# GameState.gd
class_name GameState
extends Resource

@export var sanctum_energy: int = 10
@export var max_action_point: int = 4
@export var current_action_point: int = 4
@export var player_units: Array = []
@export var enemy_units: Array = []
@export var player_hand: Array = []
@export var player_draw_pile: Array = []
@export var player_discard_pile: Array = []
```

**资源文件示例**：`res://resources/game_state/game_state.tres`

## 五、核心流程

### 5.1 游戏启动流程

1. 加载配置文件
2. 初始化GameManager
3. 显示主菜单

### 5.2 战斗流程

1. 初始化战斗场景
2. 部署初始敌人
3. 玩家回合：
   - 行动点重置
   - 法强值结算
   - 玩家操作（位移、抽牌、激活、技能发动）
   - 回合结束
4. 敌方回合：
   - 敌人行动判定
   - 敌人行动执行
   - 回到玩家回合
5. 胜负判定：
   - 所有敌人死亡 → 胜利
   - 圣物能量为0 → 失败

### 5.3 卡牌使用流程

1. 玩家选择卡牌
2. 校验使用条件（行动点、法强、时机）
3. 执行卡牌效果
4. 更新游戏状态
5. 更新UI显示

## 六、技术实现要点

### 6.1 Godot实现

- 使用Godot 4.x引擎
- 采用GDScript编写脚本
- 使用场景树组织游戏对象
- 使用信号系统实现模块间通信
- 使用Resource类实现数据资源管理：
  - 创建自定义Resource类（CardResource、EnemyResource、GameState等）
  - 在编辑器中创建和编辑资源文件（.tres）
  - 使用load()函数加载资源
  - 使用@export注解实现属性可视化编辑

### 6.2 性能优化

- 使用对象池复用卡牌和敌人节点
- 优化渲染性能，减少Draw Call
- 合理使用异步加载，避免卡顿

### 6.3 测试策略

- 单元测试核心功能
- 集成测试模块间交互
- 性能测试战斗场景

## 七、开发优先级

1. **核心战斗机制**：实现基本的回合流程和卡牌使用
2. **UI系统**：实现基本的游戏界面和交互
3. **敌人AI**：实现简单的敌人行为
4. **数据管理**：实现配置文件加载和游戏状态管理
5. **优化和测试**：提升性能和稳定性

## 八、总结

本架构设计针对《彼岸史诗：战团》的简单Demo版本，采用简化的模块化设计，优先实现核心游戏机制，确保游戏性验证的顺利进行。通过数据驱动和模块化设计，便于后续功能的扩展和修改，为完整游戏的开发奠定基础。
