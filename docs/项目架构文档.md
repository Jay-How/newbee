# 项目架构文档

## 1. 项目根目录结构概览

```
newbee/
├── assets/                # 游戏资产
├── src/                   # 源代码
├── data/                  # 数据文件
├── docs/                  # 文档
└── project.godot          # 项目配置文件
```

## 2. 各主要文件夹的具体职责说明

### 2.1 assets/ 文件夹
- **职责**：存放游戏中所有的美术、音效、模型等资源文件
- **用途**：为游戏提供视觉和听觉效果，增强游戏体验
- **管理原则**：按资源类型分类存储，便于管理和加载

### 2.2 src/ 文件夹
- **职责**：存放游戏的所有源代码文件
- **用途**：实现游戏的核心逻辑和功能
- **管理原则**：按功能模块分类存储，确保代码结构清晰

### 2.3 data/ 文件夹
- **职责**：存放游戏的所有数据文件
- **用途**：存储卡牌、技能、效果等游戏数据，实现数据与逻辑分离
- **管理原则**：按数据类型分类存储，便于数据的维护和更新

### 2.4 docs/ 文件夹
- **职责**：存放项目的所有文档文件
- **用途**：记录项目的设计、开发、测试等相关信息
- **管理原则**：按文档类型分类存储，确保文档的完整性和可访问性

## 3. 每个文件夹下应包含的文件类型、命名规范及存放标准

### 3.1 assets/ 文件夹

#### 3.1.1 子文件夹结构
```
assets/
├── cards/             # 卡牌资源
│   ├── unit/          # 单位卡
│   ├── strategy/      # 策略卡
│   └── enemy/         # 敌方卡
├── ui/                # UI资源
│   ├── icons/         # 图标
│   ├── panels/        # 面板
│   └── fonts/         # 字体
├── models/            # 3D模型
├── sounds/            # 音效
│   ├── music/         # 背景音乐
│   └── sfx/           # 音效
└── textures/          # 纹理
```

#### 3.1.2 命名规范
- **文件命名**：使用小写字母，单词间用下划线分隔，如 `temple_knight.png`
- **文件夹命名**：使用小写字母，单词间用下划线分隔，如 `unit_cards`
- **资源类型**：
  - 卡牌资源：PNG格式，尺寸为200x300px，透明背景
  - UI资源：PNG格式，根据具体需求确定尺寸
  - 音效资源：背景音乐为MP3格式，音效为WAV格式
  - 模型资源：FBX格式，低多边形

### 3.2 src/ 文件夹

#### 3.2.1 子文件夹结构
```
src/
├── core/              # 核心系统
│   ├── CardSystem.gd  # 卡牌系统
│   ├── BattleSystem.gd # 战斗系统
│   ├── PlayerSystem.gd # 玩家系统
│   ├── AISystem.gd    # AI系统
│   ├── DataManager.gd # 数据管理
│   ├── KeywordSystem.gd # 词条系统
│   └── EffectSystem.gd # 效果系统
├── scenes/            # 场景
│   ├── BattleScene.tscn # 战斗场景
│   ├── MapScene.tscn  # 地图场景
│   └── MenuScene.tscn # 菜单场景
├── nodes/             # 节点
│   ├── Card.gd        # 卡牌节点
│   ├── UnitCard.gd    # 单位卡节点
│   ├── StrategyCard.gd # 策略卡节点
│   └── EnemyCard.gd   # 敌方卡节点
└── utils/             # 工具类
    ├── MathUtil.gd    # 数学工具
    └── StringUtil.gd  # 字符串工具
```

#### 3.2.2 命名规范
- **文件命名**：使用 PascalCase 命名法，如 `CardSystem.gd`
- **文件夹命名**：使用小写字母，单词间用下划线分隔，如 `core`
- **类命名**：使用 PascalCase 命名法，如 `class_name CardSystem`
- **函数命名**：使用 snake_case 命名法，如 `func load_card_data() -> void`
- **变量命名**：使用 snake_case 命名法，如 `var card_data: Dictionary`

## 4. 跨文件夹的依赖关系和交互规则

### 4.1 依赖关系
- **src/core/ 依赖 data/**：核心系统需要加载和使用 data 目录中的数据文件
- **src/scenes/ 依赖 src/core/**：场景需要使用核心系统提供的功能
- **src/nodes/ 依赖 src/core/**：节点需要使用核心系统提供的功能
- **src/utils/ 被其他模块依赖**：工具类被其他模块使用

### 4.2 交互规则
- **数据流向**：data → src/core → src/scenes/src/nodes
- **调用方式**：使用 Godot 的 Resource 系统加载数据，使用信号机制进行模块间通信
- **依赖注入**：尽量使用依赖注入的方式，减少模块间的紧耦合
- **避免循环依赖**：确保模块间的依赖关系是单向的，避免循环依赖

## 5. 特殊文件的存放位置及处理方式

### 5.1 项目配置文件
- **文件**：`project.godot`
- **存放位置**：项目根目录
- **处理方式**：由 Godot 引擎自动管理，包含项目的基本配置信息

### 5.2 资源元文件
- **文件**：`*.gd.uid` 文件
- **存放位置**：与对应的 `.gd` 文件同一目录
- **处理方式**：由 Godot 引擎自动生成和管理，无需手动修改

### 5.3 场景文件
- **文件**：`*.tscn` 文件
- **存放位置**：`src/scenes/` 目录
- **处理方式**：使用 Godot 编辑器创建和编辑，包含场景的节点结构和属性

### 5.4 数据资源文件
- **文件**：`*.tres` 文件
- **存放位置**：`data/` 目录下的对应子目录
- **处理方式**：使用 Godot 编辑器创建和编辑，包含游戏数据

## 6. 最佳实践

### 6.1 文件组织原则
- **单一职责**：每个文件和模块只负责一个功能
- **模块化**：将相关功能组织到同一个模块中
- **可维护性**：代码和数据的组织应便于维护和更新
- **可扩展性**：设计应考虑未来的扩展需求

### 6.2 命名规范
- **一致性**：整个项目使用一致的命名规范
- **可读性**：命名应清晰易懂，反映文件或变量的用途
- **避免冲突**：确保命名不会与 Godot 内置函数或变量冲突

### 6.3 代码规范
- **注释**：为关键代码添加注释，解释代码的功能和逻辑
- **格式**：保持代码格式一致，使用适当的缩进和空行
- **错误处理**：添加适当的错误处理，提高代码的健壮性
- **性能**：考虑代码的性能，避免不必要的计算和资源消耗

### 6.4 版本控制
- **提交规范**：使用清晰的提交信息，描述代码变更的内容和原因
- **分支管理**：使用分支管理不同的开发任务和版本
- **标签**：使用标签标记重要的版本和发布
